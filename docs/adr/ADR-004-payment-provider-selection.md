# ADR-004: 決済プロバイダ選定 - つながる募金の採用

## Status
Accepted

## Context

タダカヨ QRチャリティ・コネクトでは、QRコードを起点としたPayPay/楽天ペイでの寄付決済を想定していた。当初は以下のアーキテクチャを計画：

- Cloud Run: APIエンドポイント、決済遷移、Webhook受信
- Firestore: 寄付履歴・決済ステータス保存
- PayPay for Developers / 楽天ペイ API: 決済連携

しかし、2026年1月の調査で以下の事実が判明：

1. **PayPay for Developers は寄付目的での利用が規約違反**
   - 公式FAQ: 「商取引ではない寄付や募金、投げ銭など一部NG商材」と明記
   - QRコードのSNS・Web掲載も規約違反（店舗加盟店の場合）

2. **楽天ペイはNPO向け寄付決済に非対応**
   - 楽天クラッチ募金は楽天運営のプラットフォームであり、NPOが自由に導入できるものではない
   - 楽天ペイ（オンライン決済）での寄付導入に関する公開情報なし

3. **PayPayで寄付を実現する方法は限定的**
   - PayPay法人ビジネスアカウント直接契約: 厳重な審査（対面調査）、約20団体のみ導入
   - つながる募金経由: PayPay対応（2025年3月〜）、720団体以上が利用
   - コングラント経由: PayPay対応、月額費用あり

## Decision

**ソフトバンク「つながる募金」を採用する。**

### 選定理由

| 評価軸 | つながる募金 | PayPay直接契約 | コングラント |
|--------|-------------|---------------|-------------|
| コスト | ◎ 2.4%のみ | ○ 要確認 | △ 月額4,000円〜+4.4% |
| 導入容易性 | ◎ 約1ヶ月 | × 厳重審査 | ○ 2〜4週間 |
| 審査リスク | ◎ 低 | × 高 | ○ 中 |
| 開発工数 | ◎ ほぼゼロ | ○ ゼロ | △ 設定作業あり |

### アーキテクチャ変更

**Before:**
```
QRコード → Cloud Run → PayPay API → Firestore
```

**After:**
```
QRコード → 既存HP → つながる募金 → PayPay決済
```

GCPインフラ（Cloud Run, Firestore, Secret Manager, Cloud NAT）は**不要**となった。

## Consequences

### メリット
- 開発コストほぼゼロ（既存HPにリンク追加のみ）
- 月額固定費なし（手数料2.4%のみ）
- 審査が比較的通りやすい（720団体以上の実績）
- PayPay以外にクレカ・携帯料金合算にも対応

### デメリット
- 寄付UXがやや煩雑（QR→Webページ→PayPay決済、4〜5タップ）
- 領収書は自動発行されない（自前で発行が必要）
- 寄付履歴のリアルタイム取得はできない（管理画面で確認）
- つながる募金のページデザインはカスタマイズ不可

### 将来の移行パス
- 寄付規模拡大時（年間50万円超）: コングラントへの移行を検討
- 寄付UX改善が必要な場合: PayPay法人ビジネスアカウント直接契約を再検討

## Alternatives

### 1. PayPay法人ビジネスアカウント直接契約
- QRスキャン→即PayPay決済の最良UX
- しかし審査が厳しく（対面調査）、認定NPO申請中の段階では不確実性が高い
- 将来的に認定取得後、再検討の余地あり

### 2. コングラント経由
- 領収書自動発行、寄付者CRM機能が充実
- 月額4,000円〜の固定費が発生
- 寄付規模が小さい初期段階ではコスパが悪い

### 3. PayPay for Developers（API直接連携）
- 当初想定していた方法
- **寄付目的での利用は規約違反のため採用不可**

### 4. 楽天ペイ
- NPO向け寄付決済に対応していないため採用不可
